Incremental ETL Pipeline with Star Schema

Designing a complete data warehouse solution using Amazon e-commerce sales data with Python, PostgreSQL, and incremental loading techniques.

By K. Asritha • April 2026 • Data Warehouse Course Project

Introduction

Data warehouses are used to store historical data in a structured format so that organizations can perform reporting, analytics, and business decision making. In this project, I developed a complete ETL pipeline using Amazon sales data and loaded it into PostgreSQL using a star schema model.

The main objective was not only to load data once, but also to implement incremental loading, where future executions process only newly arrived records. This reflects how modern production systems reduce processing time and improve efficiency.

Project Objectives

  • Collect and analyze Amazon sales CSV datasets
  • Clean and transform raw records using Python
  • Design a star schema data warehouse
  • Load data into PostgreSQL
  • Implement incremental ETL using watermark logic
  • Generate business insights using SQL queries

System Architecture

CSV Files → Python (Pandas) → Data Cleaning → PostgreSQL Warehouse → SQL Analysis / Dashboard

The project follows a standard ETL architecture. Source CSV files are extracted using Python, transformed into clean structured data, and loaded into PostgreSQL tables. Once the warehouse is created, SQL queries are used for analysis.

Star Schema Design

A star schema consists of one central fact table connected to multiple dimension tables. This design improves analytical query performance and keeps data organized.

Fact Table

  • fact_sales – stores transaction level sales records

Dimension Tables

  • dim_date
  • dim_product
  • dim_location
  • dim_order_status
  • dim_fulfilment

This model allows easy reporting such as monthly sales, category revenue, state-wise performance, and fulfilment comparisons.

ETL Pipeline

1. Extract

Raw CSV files were loaded into Pandas DataFrames. 

import pandas as pd
df = pd.read_csv("Amazon Sale Report.csv", low_memory=False)

2. Transform

  • Renamed messy columns
  • Converted text dates into date format
  • Handled missing amount values
  • Removed duplicates
  • Created dimension tables
  • Generated surrogate keys

3. Load

Dimension tables were loaded first, followed by the fact table to maintain foreign key integrity.

Incremental Loading using Watermark

Loading the full dataset repeatedly wastes time. Therefore, I implemented a watermark strategy.

Run 1 → Load all records
Run 2 → Load rows where order_date > last_loaded_date
Run 3+ → Continue loading only new records

The latest processed date is stored in an etl_watermark table. 

SELECT last_loaded_date FROM etl_watermark;

WHERE order_date > watermark_date

This significantly reduces load time for future executions.

Business Insights

After loading data into the warehouse, analytical SQL queries were executed.

  • Maharashtra generated highest sales revenue
  • April 2022 recorded peak monthly sales
  • Set category had highest revenue
  • B2C orders were higher than B2B orders
  • Cancelled orders formed a notable percentage
SELECT state, SUM(amount_inr)
FROM fact_sales
GROUP BY state
ORDER BY SUM(amount_inr) DESC;
