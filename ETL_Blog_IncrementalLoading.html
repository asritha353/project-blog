Incremental ETL Pipeline with Star Schema Design

How I built a production-grade data warehouse on Amazon e-commerce data — going beyond standard batch loading with a watermark-based incremental strategy.

📅 April 2026 ⏱ 10 min read 🔧 Technical Deep Dive 📊 128K+ Records

Most data warehouse courses teach you the fundamentals: build a star schema, write an ETL, load your data. And that's valuable. But when I sat down to actually implement this on a real dataset — 128,000+ Amazon India sales records — I hit a wall that the textbook didn't prepare me for.

What happens when the data grows? What happens on the second run? Do I re-load all 128,000 rows every single time? That would be wasteful, fragile, and nothing like how real production pipelines work.

That question led me to implement incremental loading with a watermark strategy — a technique that goes well beyond the standard syllabus. This post walks through the entire project: why I made each design decision, how the star schema was constructed, and how the incremental ETL actually works.

The Problem with Standard Batch ETL

A typical introductory ETL pipeline works like this: read the CSV, clean the data, insert into the database. Simple. But it hides a serious flaw — every time the pipeline runs, it reloads everything.

❌ Standard Batch ETL

Re-loads all rows every run

Scales linearly with data growth

No checkpoint — crashes restart from zero

Wastes compute on unchanged records

Cannot run multiple times safely

✅ Incremental ETL (This Project)

Only loads records newer than last run

Scales with new data volume, not total

Watermark checkpoint survives crashes

Idempotent — safe to re-run

Production-grade pattern used at scale

For our 128,000-row Amazon dataset, this distinction might seem minor — but the design principle scales to billions of rows. Companies like Amazon, Swiggy, and Flipkart run incremental pipelines because reloading petabytes of data nightly is simply not feasible.

Designing the Star Schema

The central question in dimensional modelling is: what is the business event I want to measure? In this project, it's each individual order line — a customer purchasing a product on a specific date, shipped to a specific location, with a specific fulfilment method.

Why a Star Schema?

A star schema separates facts (measurable numbers like quantity and revenue) from dimensions (descriptive context like product category, date, location). This separation makes analytical queries dramatically faster and more readable than a flat denormalised table.

The Final Schema: 1 Fact + 5 Dimensions

TableTypeKey ColumnsWhy It Exists
fact_salesFactquantity, amount_inr, is_b2b, load_timestampOne row per order. Holds measures + foreign keys to all dims.
dim_dateDimensiondate_key, full_date, day, month, quarter, year, is_weekendEnables time-series analysis, quarter comparisons, weekend vs weekday splits.
dim_productDimensionsku, style, category, size, asinCategory-level sales breakdowns, size analysis.
dim_locationDimensioncity, state, postal_code, countryGeographic sales distribution — which states generate most revenue.
dim_order_statusDimensionstatus_name, status_categorySimplifies raw status strings into Completed / In Transit / Cancelled.
dim_fulfilmentDimensionfulfilment_type, service_level, fulfilled_byCompare Merchant vs Amazon-fulfilled orders.
💡 Design Decision: Surrogate Keys

Every dimension table uses a surrogate primary key (auto-increment integer) rather than natural keys like SKU or order ID. Natural keys can change, contain special characters, or be inconsistent across sources. Surrogate keys are stable, compact, and join faster — this is standard practice in production data warehouses.

The ETL Pipeline

The pipeline is implemented in Python using Pandas for transformation and SQLAlchemy for database operations. Each phase is in its own module for maintainability.

Phase 1: Extract

Reading raw CSV files into DataFrames. The main file requires low_memory=False to handle the mixed-type columns: 

# etl/extract.py
import pandas as pd

def extract_amazon_sales():
    df = pd.read_csv('data/raw/Amazon Sale Report.csv',
                      low_memory=False)
    print(f'Extracted {len(df)} rows')
    return df

Phase 2: Transform

This is the most complex phase. Twenty-three raw columns needed renaming, dates needed parsing, nulls needed handling, and five dimension DataFrames needed to be built from the cleaned main DataFrame.

Key transformations:

  • Rename all 23 columns to clean snake_case names
  • Convert date strings: pd.to_datetime(df['order_date'], format='%m-%d-%y', errors='coerce')
  • Fill missing amount_inr with 0; fill missing cities with 'UNKNOWN'
  • Create status_category (Completed / In Transit / Cancelled) from raw status strings
  • Build dimension DataFrames with deduplicated rows and assigned surrogate keys
# Build dim_date from the cleaned DataFrame
def build_dim_date(df):
    dates = df['order_date'].drop_duplicates()
    dim = pd.DataFrame({'full_date': dates})
    dim['date_key']   = dim['full_date'].dt.strftime('%Y%m%d').astype(int)
    dim['day']       = dim['full_date'].dt.day
    dim['month']     = dim['full_date'].dt.month
    dim['quarter']   = dim['full_date'].dt.quarter
    dim['year']      = dim['full_date'].dt.year
    dim['is_weekend']= dim['full_date'].dt.dayofweek >= 5
    return dim

Phase 3: Load

Loading order is critical: dimensions must be fully inserted before the fact table, because the fact table's foreign keys reference dimension primary keys. Loading in the wrong order triggers FK constraint violations.

1

dim_date → dim_product → dim_location → dim_order_status → dim_fulfilment

All five dimensions loaded first with SQLAlchemy to_sql(), chunk size 1000.

2

Join surrogate keys onto main DataFrame

Merge each dim's key column back onto the cleaned data using pd.merge().

3

Load fact_sales

Insert the final fact DataFrame — now containing FK references — in chunks of 500 rows.

Incremental Loading — The Advanced Part

This is the technique that goes beyond what's typically covered in a data warehouse course — and it's what I'm most proud of in this project.

⭐ Beyond the Syllabus

Standard courses teach full ETL loads — process everything, every time. Incremental loading is what production pipelines actually use. It's the difference between a student project and a real engineering solution.

The Watermark Strategy

A watermark is a saved checkpoint that records how far the ETL has processed. In this project, it's the maximum order_date from the last successful pipeline run, stored in a dedicated table: 

-- etl_watermark table
CREATE TABLE etl_watermark (
    table_name       VARCHAR(100) PRIMARY KEY,
    last_loaded_date DATE,
    last_run_at      TIMESTAMP
);

How Each Run Works

RunWhat HappensRecords Processed
Run 1 (First time)No watermark found → load ALL historical data → save watermark = max date128,000+ rows
Run 2 (Next day)Read watermark → filter WHERE order_date > watermark → load only new rows → update watermark~500 new rows
Run 3+ (Ongoing)Same pattern — only new data since last checkpoint is ever touched0 to N new rows
 
# etl/incremental.py
def incremental_load_facts(df_clean):
    watermark = get_watermark('fact_sales')

    if watermark is None:
        # First run: load everything
        new_data = df_clean
    else:
        # Subsequent runs: only new records
        new_data = df_clean[
            df_clean['order_date'].dt.date > watermark
        ]

    if len(new_data) > 0:
        load_fact(new_data)
        max_date = new_data['order_date'].max().date()
        update_watermark('fact_sales', max_date)
    else:
        print('No new data. Pipeline is up to date.')

The get_watermark() and update_watermark() functions use SQLAlchemy's text() with an ON CONFLICT DO UPDATE (PostgreSQL upsert) to atomically update the checkpoint. This ensures the watermark is always in sync with what was actually loaded.

💡 Why This Matters

If the pipeline crashes mid-run, the watermark is not updated. The next run will re-process the same records — which is safe because of idempotent loading — and then save a valid checkpoint. The watermark only advances when a run completes successfully.

Business Insights from the Warehouse

Once the warehouse was loaded, I ran SQL queries against the star schema to extract real business insights. This is the payoff for all the engineering work — the schema makes these queries simple and fast. 

-- Revenue by product category
SELECT p.category,
       SUM(f.amount_inr) AS total_revenue,
       COUNT(*) AS orders
FROM fact_sales f
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY p.category
ORDER BY total_revenue DESC;

Key findings from the data:

  • Set was the highest-revenue category, followed by Kurta and Western Dress
  • Maharashtra accounted for nearly 30% of all domestic orders
  • Approximately 41% of orders were in a Cancelled state — a significant operational signal
  • B2B orders made up only ~8% by volume but represented a disproportionate share of high-value transactions
  • April 2022 was the peak sales month in the dataset

Challenges and What I Learned

Data Quality is the Real Work

Roughly 70% of the implementation time was spent on the Transform phase. The raw data had 23 messy column names, dates in a non-standard format, missing monetary values, and city names in inconsistent cases. Real data is never clean — the Transform phase is where most real ETL engineering effort goes.

Foreign Key Loading Order

On my first attempt, I loaded the fact table before the dimensions were fully populated. PostgreSQL immediately threw a foreign key constraint violation. This forced me to think carefully about loading order and understand why constraints exist — not just how to work around them.

Incremental vs Idempotent

Incremental loading and idempotent loading are related but different concepts. Incremental means "only new data." Idempotent means "running it twice produces the same result as running it once." My implementation aims for both — the watermark ensures only new data is processed, and because we check for duplicates, re-running is safe.

Conclusion

Building this project end-to-end — from a raw Kaggle CSV to a queryable star schema with production-grade incremental loading — gave me a much clearer picture of what data warehouse engineering actually involves.

The standard ETL you learn in a course is a starting point. What makes a pipeline production-ready is the infrastructure around it: checkpointing, idempotency, error handling, data quality validation, and load order awareness. This project implements the foundational version of all of these.

The incremental watermark strategy in particular is something I'd never encountered in coursework — but it's standard practice in every data engineering team I've read about. Adding it to this project means the pipeline is not just academically correct, but architecturally sound.
